# Crawlab Read-Only Discovery Design

## Goal

Собрать безопасный discovery-пакет по двум указанным `project id` в Crawlab, не строя продукт и не проектируя финальную архитектуру, а только получая и нормализуя артефакты, на основании которых другой LLM сможет сформулировать требования и MVP.

Целевые проекты:
- `66a25c4d116add6c8f235756`
- `66a384f5116add6c8f235803`

## Non-Goals

- Не строить read-only monitoring product.
- Не проектировать финальную архитектуру системы хранения, UI или алертинга.
- Не выполнять управляющие действия в Crawlab.
- Не делать массовый сбор логов или полный поиск по логам.
- Не выдумывать endpoint'ы, поля ответа или смысл параметров.

## Critical Constraints

- Использовать только `GET`-запросы.
- Любые `POST`, `PUT`, `PATCH`, `DELETE` запрещены.
- Все сетевые вызовы должны идти только через единый transport layer.
- Transport layer должен hard-block'ить любые HTTP-методы кроме `GET` до отправки запроса.
- Доступ к Crawlab брать только из переменных окружения `CRAWLAB_BASE_URL` и `CRAWLAB_API_KEY`.
- Нельзя читать `.env` как файл, печатать API key, сохранять API key в артефакты или логи.
- Нагрузка на сервер должна быть низкой: только последовательные запросы, небольшие выборки, точечное углубление, без polling.
- Если endpoint не подтвержден или ответ непонятен, фиксировать `observation` или `unknown`, не домысливать.

## Product Context To Preserve

- Это read-only инструмент операционного мониторинга, а не клиент управления Crawlab.
- Основная ценность: понимать, какие задачи выглядят нормальными, какие завершились, какие подозрительны и где вероятно требуется вмешательство.
- Технический status задачи не равен семантической корректности результата.
- Базовая единица наблюдения: `spider + schedule_id + normalized params`.
- Для анализа важны задачи, расписания, логи, параметры запуска, длительность, node, project.
- `users` для MVP не приоритетны.
- Пустой лог считается плохим сигналом.
- В MVP не нужен полный поиск по логам; нужны причины тревоги, ключевые фрагменты и `tail`.
- История потом будет храниться локально за месяц, но этот discovery не реализует само хранилище истории.
- Часто встречающиеся параметры: `-sp`, `-fp`.
- Реже встречающиеся параметры: `-pc`, `-as`, `-tags`.
- При ручных перезапусках могут использоваться `-sp`, `-fp`, `-spi`.
- `debug/test` должны считаться исключенными из production-аналитики.
- `proxy_country` и `proxy_vpn_country` меняют способ исполнения, но не бизнес-смысл задачи.

## Confirmed Inputs And Facts

### Environment

- `CRAWLAB_BASE_URL` доступен в окружении процесса.
- `CRAWLAB_API_KEY` доступен в окружении процесса.

### Documented Crawlab GET Endpoints

Подтверждено документацией Crawlab через Context7:

- `GET /api/projects`
- `GET /api/spiders`
- `GET /api/schedules`
- `GET /api/tasks`
- `GET /api/tasks/{id}`
- `GET /api/tasks/{id}/logs`
- `GET /api/nodes`
- `GET /api/nodes/{id}`

Для текущей инсталляции Crawlab эти endpoint'ы все равно должны считаться instance-level observation только после успешного фактического `GET`-ответа.

Все остальные endpoint'ы и query-параметры считаются неподтвержденными, пока не подтверждены документально или фактическим ответом API.

### Local Code Artifacts Relevant To Parameter Discovery

В локальных wheel-артефактах найден код, который может содержать факты о параметрах запуска пауков:

- `crawlib/crawllib-2.12.3-py3-none-any.whl`:
  - `crawllib/crawlab.py`
- `crawlib/cdek_crawlab_module-2.2.1-py3-none-any.whl`:
  - `cdek_crawlab_module/spider_manager.py`

Эти источники должны использоваться только как дополнительный источник фактов о параметрах и поведении запуска, а не как основание для домысливания полей Crawlab API.

## Recommended Discovery Approach

Использовать staged discovery по двум проектам с жестким контролем нагрузки.

### Stage 1: Project-Scoped Reconnaissance

Для каждого целевого `project id`:

- подтвердить, что проект наблюдается в доступных данных;
- получить минимальные базовые поля проекта;
- не выполнять широкий обзор всех сущностей без причины.

Если проект нельзя надежно привязать по подтвержденному endpoint'у или полям ответа, это фиксируется как `unknown`.

### Stage 2: Spiders

- Получить список пауков.
- Зафиксировать, какие пауки относятся к каждому целевому проекту, только на основании полей ответа.
- Сохранить raw JSON и нормализованный список важных полей.

### Stage 3: Schedules

- Получить schedules.
- Связать schedule с spider и project только по наблюдаемым данным.
- Зафиксировать, какие поля выглядят стабильными и полезными для future monitoring.

### Stage 4: Recent Tasks

- Получить последние задачи.
- Сначала брать небольшую выборку.
- Для каждого целевого паука зафиксировать доступные поля: статус, timestamps, duration, node, schedule relation, command/args, log availability и другие явно наблюдаемые поля.

### Stage 5: Expanded Task Sampling For Important Spiders

Для важного паука можно расширить выборку до `10-20` последних задач, если это реально дает новые паттерны.

Паук считается важным, если выполнено хотя бы одно условие:

- есть schedule;
- есть недавние задачи;
- есть разнообразие статусов;
- есть различающиеся паттерны параметров;
- есть признаки ручных перезапусков или подозрительных завершений.

### Stage 6: Targeted Task Details, Logs And Nodes

Только точечно и только для задач, которые помогают собрать репрезентативные кейсы:

- нормальный успешный запуск;
- `failed`;
- `finished`, но подозрительный;
- `long-running`, если такой статус или явный длительный кейс реально существует;
- manual rerun candidate;
- всплеск HTTP-ошибок, если такой паттерн действительно найден.

Для каждой выбранной задачи:

- при необходимости получить task details;
- при необходимости получить task logs;
- при необходимости получить node details.

Если нужный тип кейса не найден, это должно попасть в `docs/discovery/open-questions.md`, а не подменяться выдуманным примером.

## Low-Load Rules

- Только последовательные API-вызовы.
- Никаких параллельных сетевых запросов к Crawlab.
- Небольшие `page/size` на первом проходе.
- Углубляться только если предыдущий ответ подтвердил полезность endpoint'а.
- Между запросами использовать небольшой throttle.
- Не тянуть массово все логи или все task details.
- Если данных уже достаточно для артефактов по пауку, дальнейшее углубление не делать.

## Implementation Design

### `collector/config.py`

Ответственность:

- читать `CRAWLAB_BASE_URL` и `CRAWLAB_API_KEY` из окружения;
- валидировать обязательные env vars;
- не печатать секреты.

### `collector/transport.py`

Единый сетевой слой. Должен:

- поддерживать только `GET`;
- блокировать любой другой HTTP-метод до отправки запроса;
- формировать абсолютный URL от `CRAWLAB_BASE_URL`;
- добавлять `Authorization` header;
- работать только последовательно;
- поддерживать throttle между вызовами;
- возвращать статус, тело ответа и метаданные вызова;
- не логировать и не сохранять API key.

Никакой другой модуль не должен делать HTTP-вызовы напрямую.

### `collector/discovery.py`

Оркестратор staged discovery. Должен:

- принимать два `project id`;
- запускать этапы discovery в фиксированном порядке;
- прекращать углубление, если endpoint не подтвержден или не добавляет полезных артефактов;
- помечать результаты как `fact`, `hypothesis` или `unknown`.

### `collector/raw_store.py`

Ответственность:

- писать сырые ответы в `docs/discovery/raw/...`;
- разделять артефакты по сущностям и scope;
- не смешивать несколько ответов в один файл без необходимости.

### `collector/normalize.py`

Ответственность:

- выделять ключевые поля проектов, пауков, schedules, задач и nodes;
- строить сырое и нормализованное представление параметров запуска;
- формировать базовую единицу наблюдения: `spider + schedule_id + normalized params`;
- явно сохранять неизвестность там, где нормализация неполна.

### `collector/log_analysis.py`

Ответственность:

- извлекать lightweight-признаки логов;
- не выполнять product-like полнотекстовый поиск;
- сохранять только то, что нужно для discovery-пакета.

Извлекаемые признаки:

- `traceback` / fatal;
- timeout;
- `403/404/5xx`;
- `found` / `written`;
- наличие итогового summary;
- приблизительная длина лога;
- `tail`;
- есть ли признаки прогресса;
- стабильно повторяющиеся фрагменты;
- нестабильные фрагменты.

### `collector/reporting.py`

Ответственность:

- собрать итоговые markdown-документы;
- отделять `fact`, `hypothesis` и `unknown`;
- не заполнять пустоты догадками.

## Output Package

Нужно получить следующие файлы:

- `docs/discovery/api-map.md`
- `docs/discovery/entity-summary.md`
- `docs/discovery/parameter-taxonomy.md`
- `docs/discovery/log-patterns.md`
- `docs/discovery/open-questions.md`

И директории с артефактами:

- `docs/discovery/raw/...`
- `docs/discovery/normalized/...`

## Raw Artifact Expectations

В `docs/discovery/raw/` нужно сохранять:

- примеры raw JSON по подтвержденным endpoint'ам;
- отдельные raw snapshots по проектам;
- списки пауков;
- списки schedules;
- recent tasks;
- выбранные task details;
- выбранные log responses;
- выбранные node responses.

Имена файлов должны быть человекочитаемыми и показывать контекст: проект, сущность, endpoint, sample scope.

## Normalized Artifact Expectations

В `docs/discovery/normalized/` нужно сохранять подготовленные срезы:

- список проектов и релевантных полей;
- список пауков целевых проектов;
- schedules по паукам;
- срезы задач по паукам;
- ключевые task fields;
- нормализованные параметры запуска;
- признаки manual rerun vs schedule-driven behavior;
- node fields, которые реально могут влиять на анализ.

## Parameter Taxonomy Requirements

Для параметров запуска нужно сохранить:

- raw representation;
- normalized representation;
- фактически наблюдаемые варианты;
- классификацию параметров по ролям:
  - `identity candidate`
  - `shard/scope candidate`
  - `execution modifier`
  - `non-production flag`
  - `rerun/recovery indicator`

Ожидаемые специальные правила:

- `proxy_country` и `proxy_vpn_country` считать `execution modifier`, а не identity.
- `debug/test` относить к `non-production flag`.
- `-sp`, `-fp`, `-spi`, `-pc`, `-as`, `-tags` сначала фиксировать как наблюденные параметры, а потом классифицировать только при наличии подтверждающих артефактов.

## Interpretation Rules

### Fact

Считается фактом только то, что подтверждено хотя бы одним из источников:

- фактический ответ Crawlab API;
- документация Crawlab, подтвержденная через Context7;
- локальный код/библиотека, если речь идет о параметрах запуска или вспомогательной логике пауков.

### Hypothesis

Считается гипотезой то, что выглядит повторяющимся паттерном, но пока недостаточно подтверждено одним артефактом или отсутствует объяснение поля.

### Unknown

Считается неизвестным то, что не удалось надежно подтвердить: endpoint, поле, связь сущностей, смысл параметра, тип лога, интерпретация статуса.

## Error Handling

- Любая попытка вызвать не-`GET` метод должна завершаться локальной ошибкой до сетевого вызова.
- `401`, `403`, `404`, `5xx`, schema mismatch, пустые и непонятные ответы должны фиксироваться как `observation` или `unknown`.
- Если документация и фактический ответ расходятся, приоритет у фактического ответа, а расхождение фиксируется отдельно.
- Пустой лог должен отмечаться как негативный сигнал.
- Недоступный или непонятный log endpoint не должен приводить к домысливанию `tail` или summary.

## Verification Strategy

Перед live-run нужны локальные тесты:

- тесты transport layer:
  - разрешен только `GET`;
  - другие методы блокируются;
  - API key не попадает в ошибки и логи;
- тесты нормализации:
  - raw args сохраняются без потерь;
  - normalized args строятся детерминированно;
  - `debug/test` попадают в `non-production`;
  - `proxy_country/proxy_vpn_country` не попадают в identity;
- тесты reporting:
  - `fact`, `hypothesis`, `unknown` не смешиваются;
  - пустые секции не маскируются под заполненные.

После локальных тестов нужен один осторожный live-run с throttle и небольшими лимитами.

## Definition Of Done

Discovery считается завершенным, когда:

- существует минимальный Python collector с единым GET-only transport;
- собраны raw artifacts в `docs/discovery/raw/`;
- собраны normalized artifacts в `docs/discovery/normalized/`;
- заполнены все пять markdown-документов;
- для обоих `project id` собраны spiders, schedules, recent tasks и доступные node/task/log observations в пределах лимитов;
- есть отдельные наблюдения по коду/библиотеке, где описаны параметры запуска пауков;
- пакет достаточно понятен, чтобы его можно было передать GPT Pro без дополнительной ручной расшифровки.

## Open Questions To Preserve During Implementation

Следующие вопросы заранее не решать догадками; их нужно либо подтвердить артефактами, либо оставить открытыми:

- какой endpoint или поле надежно связывает spider с конкретным project;
- какие query-параметры реально поддерживаются для project-scoped фильтрации spiders, schedules и tasks;
- где именно в API отражаются launch parameters и command/args в пригодном для нормализации виде;
- какие поля позволяют отличить schedule-driven запуск от manual rerun;
- какие статусы или комбинации полей достаточно надежно указывают на `long-running` или `finished-but-suspicious`;
- какой формат response у log endpoint на этой инсталляции Crawlab;
- какие node fields действительно коррелируют с operational analysis.
